## Documented Architectural Divergences

### 1. JSON files instead of SQLite

**What the spec says:** Phases 3–5 use a SQLite registry (`poea_registry.sqlite`) with tables `concepts`, `induction_runs`, `concept_evidence_links`, `concept_events`.

**What was implemented:** JSON file artifacts (`concept_registry.json`, `canonical_concepts.json`). The registry is rebuilt on each `consolidate` run from `raw_concepts.json`.

**Phase 6 decision:** Evidence scoring results are cached as `scored_evidence.json`. This satisfies the Phase 6 caching requirement (no re-scoring on re-runs) without introducing SQLite. Cache lookup is O(n) over evidence records and O(1) per concept lookup via set intersection, which is acceptable at MVP scale (70 records × 11 concepts).

**Why chosen over SQLite:** Introducing SQLite only for Phase 6 would create an inconsistent dual-store architecture (JSON for registry, SQLite for evidence links). A full migration to SQLite (option 3) would require re-implementing Phases 3–5. The JSON approach satisfies all documented exit criteria.

**Future note:** If Phase 9 (POE Adapter) or Phase 11 (Run Reports) reveal query-pattern requirements that JSON cannot satisfy, a migration to SQLite should be considered.

### 2. Fireworks AI instead of Anthropic SDK

**What the spec says:** `anthropic` SDK is the default LLM provider.

**What was implemented:** Fireworks AI (OpenAI-compatible endpoint), model `deepseek-v4-pro`.

**Why:** Provider decision made during Phase 2. The LLM client is provider-agnostic via the `LLMClient` protocol. Switching providers requires only changing `FireworksClient` in `src/poea/llm.py` and the model config.

**Impact on Phase 6:** Evidence scoring uses the same `FireworksClient`. Scoring calls are batched per evidence record (all active concepts in one call) to reduce API cost.

### 3. CLI command structure differs from spec

**What the spec says:**
```
poea registry init --db ...
poea registry import --db ...
poea registry list --db ...
poea registry diff --db ...
poea registry promote --db ... --auto
poea score-evidence --db ... --evidence ...
```

**What was implemented:**
```
poea consolidate               (consolidation + promotion in one step)
poea registry promote          (standalone re-promotion, JSON-based)
poea score-evidence            (Phase 6; uses --concepts instead of --db)
```

`registry init`, `registry import`, `registry list`, and `registry diff` are not implemented. These are SQLite-oriented commands from the original plan and are superseded by the JSON-file workflow.
